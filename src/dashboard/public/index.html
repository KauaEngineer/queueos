03 · How it works

The classic producer-consumer pattern

Your API enqueues jobs in Redis. Workers consume in parallel and persist results to PostgreSQL. The dashboard polls Redis stats every 2 seconds for live observability.

04 · Features

Everything you need to run jobs in production

Built-in retry policies, observability, multi-tenancy and deployment artifacts — no extra plugins required.

⚡

4 specialized queues

Dedicated pools for email, reports, image processing and notifications — each with its own concurrency, timeout and retry policy.

Per-tenant job tracking via the X-Tenant-Id header. Filter the dashboard and aggregate metrics by tenant.

06 · Developer experience

Strongly typed. No surprises.

Every job is a TypeScript interface. IDE autocomplete and compile-time checks across producers and workers. 

// Producer: enqueue from your API handler
import { enqueueEmail } from './producers/jobProducer';

async function sendWelcomeEmail(user: User) {
  await enqueueEmail({
    to: user.email,
    subject: 'Welcome to our app!',
    template: 'welcome',
    data: { name: user.name }
  }, {
    tenantId: 'acme-corp',
    priority: 1,        // 1 = highest
    delay: 5000          // optional: wait 5s
  });
}
// Worker: process jobs with progress & retry
import { Worker } from 'bullmq';
import { redisConnection } from '../config/redis';

export const emailWorker = new Worker(
  'email-sender',
  async (job) => {
    await job.updateProgress(25);
    const html = await renderTemplate(job.data);

    await job.updateProgress(60);
    await transporter.sendMail({ ...job.data, html });

    return { sent: true, ts: Date.now() };
  },
  {
    connection: redisConnection,
    concurrency: 10,
    // auto-retry with exponential backoff
  }
);
// Schedule recurring jobs via the API or dashboard UI

// 1. Via REST API
await fetch('/api/cron', {
  method: 'POST',
  headers: { 'X-Tenant-Id': 'acme' },
  body: JSON.stringify({
    queue: 'report-gen',
    pattern: '0 9 * * *',    // daily at 9am
    payload: {
      reportType: 'daily-sales',
      userId: 'admin'
    }
  })
});

// 2. Programmatically
await enqueueReport(payload, {
  repeat: { pattern: '*/15 * * * *' } // every 15min
});

10 · FAQ

Common questions

Some of what recruiters and engineers usually ask about this project.

Why BullMQ instead of plain Redis or RabbitMQ? +
BullMQ gives us retries with exponential backoff, delayed jobs, priority lanes, rate limiting and a rich event API — for free, on top of Redis. RabbitMQ is great but adds operational overhead; here Redis doubles as cache + broker.
Why not just use serverless functions / Vercel Cron? +
Serverless excels at request-response, but struggles with long-running jobs (>60s on most plans), real-time progress updates, observability and cost predictability. QueueOS workers run continuously and can process jobs of any duration with zero cold-start overhead.
Is this production-ready? +
The architecture is — BullMQ + Redis is what powers job processing at Atlassian, Microsoft and thousands of startups. This specific implementation is a portfolio project; before shipping to production you'd want test coverage, structured logging (pino), Sentry integration and stricter resource limits.
How does multi-tenancy work? +
Every API request reads X-Tenant-Id from the headers (default: "default"). Jobs carry the tenant ID in their data, workers persist it to the audit log, and dashboard queries filter by tenant. Different tenants never see each other's data.
How do I deploy this? +
Three options: docker compose up for local stack (Redis + Postgres + workers + dashboard); npm start on any Node host (Render, Railway, Fly.io) with managed Redis/Postgres; or kubectl apply -f k8s/ on any Kubernetes cluster, with HPA auto-scaling workers based on queue depth.
What's the cost to run this in production? +
Free tier: $0/month (Render free + Neon free + Upstash free, with the caveat that the web service sleeps after 15 min of inactivity). Small production: ~$15/month (DigitalOcean droplet + managed Redis + managed Postgres). Heavy workload: scales horizontally with your worker count.
